# PRISM Club Roadmap

## 1. 단계별 전략

PRISM Club은 PRISM 생태계의 “지식형 커뮤니티 레이어”로 시작한다. 처음부터 멤버십, 배지, 시즌 랭킹까지 크게 만들기보다, 예능/오프라인 놀이 주제별 Topic Hub에서 정리된 정보와 커뮤니티 대화가 함께 쌓이는 흐름을 먼저 완성한다.

핵심 세로 슬라이스:

```text
참가자/기획자 커뮤니티 선택
→ Topic Hub 선택
→ 주제 정보/이벤트/레퍼런스 확인
→ 유저 생성 방 만들기 또는 기존 방 입장
→ 대표 이벤트/레퍼런스 확인
→ 타임라인 조회
→ 이벤트 카드/레퍼런스를 첨부한 글 작성
→ 댓글/대댓글
→ 검색/저장/알림
→ 신고/운영자 처리
```

## 2. Phase 0: 설계와 기반

기간: 1주

| 작업 | 산출물 |
| --- | --- |
| 제품 범위 확정 | Topic Hub, 참가자/기획자 커뮤니티, 유저 생성 방, 게시글/댓글 중심 MVP |
| 기술 스택 확정 | 버전 표, 로컬 실행 방식 |
| 도메인 모델 확정 | Topic Hub ERD, API 초안 |
| 커뮤니티 정책 초안 | 방 규칙, 유저 생성 방 기준, 신고 사유, 스포일러, 레퍼런스, 모집 글 정책 |
| 개발 환경 구성 | monorepo, lint, format, test |

완료 기준:

1. 로컬에서 앱/백엔드/DB가 한 명령으로 실행된다.
2. OpenAPI 초안이 생성된다.
3. Space, Category, TopicHub, KnowledgeBlock, TopicSignal, Room, RoomPin, Post, Reply, EventCard, Reference, Report, UserRole migration이 생성된다.

## 3. Phase 1: Community MVP

기간: 4~6주

| 주차 | 목표 | 기능 |
| --- | --- | --- |
| 1 | 계정/프로필/구조 | 로그인 mock, 프로필, 참가자/기획자 space, Topic Hub 목록 |
| 2 | Topic Hub | 개요, 지식 블록, 관련 이벤트/레퍼런스, 관련 방 |
| 3 | 유저 생성 방 | 방 만들기, 방 유형, 대표 이벤트/레퍼런스 고정 |
| 4 | 타임라인/글쓰기 | 방 타임라인, 게시글 작성/수정/삭제, 댓글/대댓글 |
| 5 | 이벤트/레퍼런스 | Events 검색 mock, 이벤트 카드, 레퍼런스 카드 |
| 6 | 검색/알림/운영 | 지식/글 통합 검색, 정보 기여 검수, 신고, 관리자 큐, audit log |

완료 기준:

1. 사용자가 참가자 커뮤니티의 연애 콘텐츠 Topic Hub에서 주제 정보와 관련 방을 볼 수 있다.
2. 사용자가 Topic Hub의 정보 개선을 제안하고 운영자가 반영할 수 있다.
3. 사용자가 Topic Hub에서 새 방을 만들 수 있다.
4. 방에는 대표 Events/Contenido 이벤트 카드와 예능 레퍼런스 카드가 고정될 수 있다.
5. 글에는 Events/Contenido 이벤트 카드와 예능 레퍼런스 카드가 첨부될 수 있다.
6. 댓글과 대댓글이 동작한다.
7. 검색에서 지식 블록, 방, 글, 이벤트 카드, 레퍼런스, 모집 글이 함께 나온다.
8. 신고된 콘텐츠와 유저 생성 방을 운영자가 숨김/복구/삭제/병합 처리할 수 있다.

## 4. Phase 2: Events/Contenido 실제 연동

기간: 3~5주

| 영역 | 작업 |
| --- | --- |
| 이벤트 검색 | 실제 Events/Contenido API와 연결 |
| 참가 인증 | 참가 완료 사용자에게 인증 후기 태그 부여 |
| 이벤트 상세 | 이벤트 상세에 관련 Club 글 노출 |
| Topic Hub 데이터 | 관련 이벤트, 인증 후기, 인기 레퍼런스, 뜨거운 쟁점 자동 반영 |
| 호스트 답변 | 이벤트 기획자 공식 답변 표시 |
| 기획자 인증 | Event Host/Studio 기반 Verified Planner 권한 부여 |
| 모집 글 | 스태프 모집 글과 댓글 문의 흐름 |
| 분석 | 이벤트 카드 클릭률, 후기 작성률, Club→Events 전환율 |

완료 기준:

1. 실제 이벤트를 검색해 Club 글에 첨부할 수 있다.
2. 참가 인증 후기와 일반 토론 글이 구분된다.
3. 이벤트 상세에서 관련 Club 글을 확인할 수 있다.

## 5. Phase 3: 커뮤니티 성장 기능

기간: 6~8주

| 기능 | 설명 |
| --- | --- |
| 홈 개인화 | 관심 카테고리, 저장, 활동 기반 피드 |
| 인용 글 | 다른 글을 인용해 새 토론 생성 |
| 지식 데이터 고도화 | 기여 이력, 버전 관리, 출처 품질 점수 |
| 방 성장 도구 | 방 대표 자료, 방 공지, 방 소유자 관리 도구 |
| 레퍼런스 컬렉션 | 주제별 예능/게임 레퍼런스 모음 |
| 참가자 신뢰 표시 | 이벤트 참가 인증, 후기 작성 이력 |
| 기획자 커뮤니티 강화 | 스튜디오별 방, 스태프 풀, 운영 노하우 아카이브 |
| 소셜링 | 같은 관심사의 소규모 만남/번개 제안 |

## 6. Phase 4: PRISM 생태계 확장

| 연동 | 방향 |
| --- | --- |
| Studio | 기획자가 Club 토론/레퍼런스를 이벤트 기획 자료로 저장 |
| Maker | 인기 레퍼런스와 토론을 기반으로 게임 룰 초안 생성 |
| MC | 현장 진행 후 자동 후기 요청과 토론방 추천 |
| Play | 게임 세션 결과를 관련 방의 토론 카드로 연결 |
| Partner | 장소/소품/촬영 정보 공유방과 업체 카드 연결 |
| Pay | 멤버십, 전용 커뮤니티, 유료 레퍼런스 팩 |

## 7. 리스크와 대응

| 리스크 | 영향 | 대응 |
| --- | --- | --- |
| 카테고리가 너무 많아짐 | 커뮤니티가 분산됨 | 초기 카테고리는 운영자가 제한적으로 생성 |
| Topic Hub가 빈약함 | 정보 허브 가치 약화 | 운영자 seed 정보, 인기 레퍼런스 자동 노출 |
| 유저 생성 방 난립 | 검색/탐색 품질 저하 | 중복 감지, 방 병합, 비활성 방 숨김 |
| 지식 기여 품질 저하 | 잘못된 정보 축적 | 근거 요구, Curator 검수, 변경 이력 |
| 글 품질 저하/스팸 | 신뢰 하락 | 신고, 금칙어, 반복 링크 제한, 운영자 큐 |
| 저작권 이슈 | 레퍼런스 공유 제한 | 원문 복제 금지, 링크/짧은 요약 중심, 신고 처리 |
| 이벤트 후기 조작 | 사용자 불신 | 참가 인증 후기 표시, 일반 토론과 구분 |
| 기획자 전용 정보 유출 | 운영 신뢰 하락 | 서버 권한 검사, 접근 로그, 권한 회수 |
| 모집 글 안전 문제 | 사용자 피해 | 필수 정보, 신고, 운영자 검수, 외부 연락 최소화 |
| 대댓글 과열 | 운영 부담 | 차단, 신고, 알림 제한, 대댓글 깊이 제한 |
| 검색 품질 부족 | 정보 탐색 가치 하락 | 초기 FTS, 성장 후 전용 검색 엔진 검토 |

## 8. 바로 다음 작업 제안

1. 저장소 구조를 확정한다: `apps/mobile`, `apps/api`, `apps/admin`, `packages/shared`.
2. API부터 만든다: Space → Category → TopicHub → Room → Post → Reply 순서.
3. Topic Hub의 지식 블록과 관련 이벤트/레퍼런스 표시를 먼저 구현한다.
4. 유저 생성 방과 방 대표 자료 고정을 구현한다.
5. mock Events adapter로 이벤트 카드 첨부를 구현한다.
6. Flutter에서는 “참가자 커뮤니티 → Topic Hub → 방 만들기 → 타임라인 → 글 상세” 세로 슬라이스를 만든다.
7. 이후 정보 기여 검수, 기획자 전용 카테고리, 모집 글, 레퍼런스 카드, 검색, 신고/관리자 화면을 붙인다.
